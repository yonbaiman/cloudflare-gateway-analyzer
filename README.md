## 🚀 Overview / プロジェクトの概要

Cloudflare Zero Trust provides world-class security, but the Free Tier has limited log export capabilities for home users and FOSS developers.
This project bridges that gap by using the **Official GraphQL API** within its intended scope to generate structured daily summaries—helping individuals understand their own DNS traffic without the need for Enterprise-grade infrastructure.

Cloudflare Zero Trust は優れたセキュリティを提供しますが、無料版ではログの外部出力に制約があります。
本プロジェクトは、**公式の GraphQL API** を想定された範囲で利用し、日次の可視化サマリーを生成します。エンタープライズ級のインフラを必要とせず、個人が自身の DNS トラフィックを把握できるよう支援します。

> 🌱 **This is a non-commercial, privacy-first FOSS project** for personal and educational use. It has no paid plans and does not monetize any data. It is **not** a commercial alternative to Cloudflare Enterprise products.
> 本プロジェクトは個人・学習用途の**非営利・プライバシー重視のFOSS**です。有料プランはなく、いかなるデータも収益化しません。Cloudflare のエンタープライズ製品の商用代替を目的とするものでは**ありません**。

- **📊 Visualized Insights**: Daily trends via Mermaid.js charts. (時間帯ごとの推移をグラフ化)
- **📥 Efficient Data Fetching**: Smart batching logic to respect API limits. (API制限に配慮した分割取得ロジック)
- **🗄️ History Persistence**: Daily accumulation of summaries into a long-term history CSV. (日次サマリーを長期履歴CSVへ蓄積)
- **🔎 Custom Reports**: On-demand reports for any date range / location. (任意期間・ロケーションのオンデマンドレポート)
- **📍 Basic Breakdown**: View DNS traffic summaries by location and device type. (ロケーション・デバイス種別ごとのトラフィックサマリー表示)
- **🔔 Daily Alerts**: Instant summary delivery via Discord Webhooks. (Discordへの毎日の解析サマリー通知機能)
- **🔒 Security First**: Two-Repo Architecture to protect your private DNS logs. (ログの機密性を守る2リポジトリ構成)

## 🔒 Security First: Two-Repo Architecture / セキュリティ設計

DNS logs contain personal browsing history. Therefore, this tool adopts the following **separation architecture**:
DNSログには個人の閲覧履歴が含まれるため、本ツールは以下の**分離アーキテクチャ**を採用しています。

1. **Public Repo (This one)**: Manages core analysis logic. Contains no personal data. (解析ロジックのみを管理。個人データは一切含まれません。)
2. **Private Repo (Yours)**: For execution, secrets, and **private log/history storage**. (実行・シークレットの保持、および**ログ／履歴の保存**を行う実行用リポジトリです。)

> 🔐 The accumulated history (`history/dns_history.csv`) is stored **only in your private repo**, never in this public repo.
> 蓄積される履歴（`history/dns_history.csv`）は**privateリポジトリにのみ**保存され、本公開リポには置かれません。

> 💡 **Tip (advanced)**: For stricter reproducibility, you may optionally pin the public repo by a commit SHA (or a release tag) instead of `main`. (任意・上級者向け：再現性を厳密にしたい場合は、`main` の代わりにコミットSHAまたはリリースタグでピン留めすることもできます。)

## 🔁 Data Flow / データの流れ

```
Every morning (cron)
  └─ harvester.py                  fetch (last ~26h) → analyze → report → notify
       └─ dns_logs_summary.csv     hourly summary (CSV)
            └─ persist_history.py   drop boundary buckets + max-count merge + retention
                 └─ private repo: history/dns_history.csv   (accumulated, 365 days)
                      └─ custom-report.yml (on-demand)
                           reads history → re-aggregate any range/location
                           → Step Summary (Mermaid) + custom_report.csv (Artifact)
```

- The daily fetch uses a **~26h lookback window split into 6h batches** (overlapping the previous day) so that cron-time drift does not create gaps between days.
- `persist_history.py` keeps only **complete hourly buckets** (drops the partial buckets at both ends of the window) and merges with a **max-count rule**, so the accumulated history stays continuous and self-healing.
- The Cloudflare API only retains ~24h of logs, so historical range queries rely on the accumulated `dns_history.csv`.

日次取得は**直近約26時間を6時間バッチに分割**し、前日と重ねて取得します（cron 実行時刻のブレで日跨ぎに隙間が生じないようにするため）。
`persist_history.py` は**完全な時間バケットのみ**を採用し（窓両端の部分集計バケットは除外）、**Count最大優先**でマージするため、履歴は連続かつ自己修復的に保たれます。
Cloudflare API はログを約24時間しか保持しないため、過去期間の集計は蓄積した `dns_history.csv` に依存します。

## 🗂️ Project Structure / scripts 構成

| File | Role |
|---|---|
| `harvester.py` | Entry point. Orchestrates fetch → analyze → report → notify. Writes `dns_logs_summary.csv`. |
| `fetcher.py` | Fetches Gateway DNS activity via GraphQL (~26h, 6h batches, boundary-safe). |
| `analyzer.py` | Maps `resolverDecision` to Allow/Block etc. (last 24h, hourly). |
| `reporter.py` | Generates Markdown (Mermaid chart), `GITHUB_STEP_SUMMARY`, and CSV. |
| `notifier.py` | Sends the daily summary to Discord via Webhook. |
| `persist_history.py` | Accumulates daily summaries into the long-term history CSV (dedup / boundary-drop / max-count / retention). |
| `query.py` | Loads & filters the accumulated history CSV (for custom reports). |
| `analyzer_custom.py` | Variable-granularity aggregation (hourly/daily) for custom reports. |
| `report_custom.py` | Entry point for on-demand custom reports (workflow_dispatch / CLI). |
| `generate_sample.py` | Generates the public Live Demo report (no private data). |

## 🚀 Quick Start / セットアップ手順

### 1. Prepare Cloudflare / Cloudflareで準備
- **`CLOUDFLARE_API_TOKEN`**: Create a token with `Zero Trust > Read` permissions. (`Zero Trust > Read` 権限を持つトークンを作成します。)
- **`CLOUDFLARE_ACCOUNT_ID`**: Found in your dashboard. (ダッシュボードから自分のアカウントIDを取得します。)

### 2. Setup Private Repo / 実行用リポジトリの作成
1. Create a new **Private** GitHub repository. (新しく「非公開」リポジトリを作成します。)
2. Add secrets to `Settings > Secrets and variables > Actions`. (Secretsに以下の値を登録します。)
   - `CLOUDFLARE_API_TOKEN`
   - `CLOUDFLARE_ACCOUNT_ID`
   - `DISCORD_WEBHOOK_URL` (Optional / 通知機能を使用する場合)

### 3. Create Workflow / ワークフローの作成
Create `.github/workflows/monitor.yml` in your **private** repo and paste the code below.
(非公開リポジトリに `.github/workflows/monitor.yml` を作成し、以下のコードを貼り付けます)

```yaml
name: Private DNS Monitor

on:
  schedule:
    - cron: '45 22 * * *'  # 07:45 JST
  workflow_dispatch:

permissions:
  contents: write          # required to commit the history CSV

concurrency:
  group: dns-history-persist
  cancel-in-progress: false

jobs:
  analyze:
    runs-on: ubuntu-latest
    timeout-minutes: 5

    steps:
      - name: Checkout Analyzer Core (public)
        uses: actions/checkout@v4
        with:
          repository: 'yonbaiman/cloudflare-gateway-analyzer'
          ref: 'main'          # for stricter reproducibility, consider pinning to a commit SHA in production
          token: ${{ secrets.GITHUB_TOKEN }}

      - name: Checkout Private (this repo, for history storage)
        uses: actions/checkout@v4
        with:
          path: private        # IMPORTANT: separate path to avoid clobbering scripts/

      - name: Setup Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.14'

      - name: Run Harvester
        env:
          CF_API_TOKEN: ${{ secrets.CLOUDFLARE_API_TOKEN }}
          CF_ACCOUNT_ID: ${{ secrets.CLOUDFLARE_ACCOUNT_ID }}
          DISCORD_WEBHOOK_URL: ${{ secrets.DISCORD_WEBHOOK_URL }}
        run: python scripts/harvester.py

      - name: Persist history
        run: |
          python scripts/persist_history.py \
            --source dns_logs_summary.csv \
            --history private/history/dns_history.csv \
            --retention-days 365

      - name: Commit history to private repo
        working-directory: private
        run: |
          git config user.name  "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          if git status --porcelain history/dns_history.csv | grep -q .; then
            git add history/dns_history.csv
            git commit -m "chore: update DNS history ($(date -u +%Y-%m-%d))"
            git push
            echo "✅ history committed."
          else
            echo "ℹ️ No history changes to commit."
          fi

      - name: Save Secure Logs
        uses: actions/upload-artifact@v4
        with:
          name: dns-analytics-csv
          path: dns_logs_summary.csv
          retention-days: 7
```

### 4. (Optional) Custom Range Report / 任意期間レポート
Add `.github/workflows/custom-report.yml` (workflow_dispatch) in your **private** repo to run on-demand reports.
Inputs: `start_date` / `end_date` / `location` / `granularity` (`auto` | `hourly` | `daily`).
It reads the accumulated `history/dns_history.csv`, re-aggregates the requested range, and outputs a Mermaid Step Summary plus `custom_report.csv` as an artifact.

任意期間・ロケーションのレポートを実行するため、非公開リポに `.github/workflows/custom-report.yml`（workflow_dispatch）を追加します。
入力：`start_date` / `end_date` / `location` / `granularity`（`auto` | `hourly` | `daily`）。
蓄積済みの `history/dns_history.csv` を読み込んで再集計し、Mermaid付き Step Summary と `custom_report.csv`（Artifact）を出力します。

> ℹ️ This workflow checks out the public repo into `core/` and your private repo into `private/`, then stages `history/dns_history.csv` where the analyzer expects it. The four `INPUT_*` env vars map the workflow inputs to `scripts/report_custom.py`.
> このワークフローは公開リポを `core/`、privateリポを `private/` にチェックアウトし、解析コアが参照する位置に履歴CSVを配置します。`INPUT_*` の4つの環境変数が入力を `scripts/report_custom.py` に渡します。

<details>
<summary><b>📄 Example: <code>.github/workflows/custom-report.yml</code> (click to expand)</b></summary>

```yaml
name: Custom DNS Report (On-Demand)

# Manual run only. Enter the range / location from the GitHub Actions screen.
on:
  workflow_dispatch:
    inputs:
      start_date:
        description: 'Start date (JST, YYYY-MM-DD)'
        required: true
        type: string
        default: '2026-06-01'
      end_date:
        description: 'End date (JST, YYYY-MM-DD, inclusive)'
        required: true
        type: string
        default: '2026-06-07'
      location:
        description: 'Location name (empty = all)'
        required: false
        type: string
        default: ''
      granularity:
        description: 'Aggregation granularity'
        required: false
        type: choice
        options:
          - auto
          - hourly
          - daily
        default: auto

permissions:
  contents: read

jobs:
  custom-report:
    runs-on: ubuntu-latest
    timeout-minutes: 5
    steps:
      # 1. Checkout the analyzer core (public repo) into core/
      #    For stricter reproducibility, consider pinning ref to a commit SHA in production.
      - name: Checkout Analyzer Core
        uses: actions/checkout@v4
        with:
          repository: 'yonbaiman/cloudflare-gateway-analyzer'
          ref: 'main'
          path: core
          token: ${{ secrets.GITHUB_TOKEN }}

      # 2. Checkout the accumulated history (this private repo) into private/
      - name: Checkout History (private)
        uses: actions/checkout@v4
        with:
          path: private
          token: ${{ secrets.GITHUB_TOKEN }}

      - name: Setup Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.14'

      # 3. Stage the history CSV where the analyzer expects it (core/history/)
      #    If missing, the workflow only warns (report_custom.py exits with "No data").
      - name: Stage history CSV
        run: |
          mkdir -p core/history
          if [ -f private/history/dns_history.csv ]; then
            cp private/history/dns_history.csv core/history/dns_history.csv
            echo "✅ history CSV staged."
          else
            echo "⚠️ private/history/dns_history.csv not found (will report 'No data')."
          fi

      # 4. Generate the custom report (run with core/ as the working directory)
      - name: Run Custom Report
        working-directory: core
        env:
          INPUT_START:    ${{ inputs.start_date }}
          INPUT_END:      ${{ inputs.end_date }}
          INPUT_LOCATION: ${{ inputs.location }}
          INPUT_GRAN:     ${{ inputs.granularity }}
        run: python scripts/report_custom.py

      # 5. Save the generated CSV as an artifact
      - name: Upload report CSV
        uses: actions/upload-artifact@v4
        with:
          name: custom-report-csv
          path: core/custom_report.csv
          retention-days: 7
          if-no-files-found: ignore
```

</details>
